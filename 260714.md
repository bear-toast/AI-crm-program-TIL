1. 가장 먼저 — x86 어셈블리 + PE 파일 포맷

오늘 한 모든 작업의 기초입니다.
- 레지스터/스택/호출규약: __stdcall, __fastcall, __thiscall 차이 (안 그러면 디컴파일된 함수 시그니처가 왜 그렇게 생겼는지 이해 못함)
- PE 포맷: DOS 헤더, 섹션 테이블, Import/Export 테이블, RVA vs 파일 오프셋 변환 — 오늘 send() 주소 찾을 때 직접 파싱했던 그거
- 추천: "x86 Assembly" 관련 무료 자료 + PE 포맷은 corkami의 PE 다이어그램이 유명합니다

2. Ghidra 사용법 (디컴파일러 읽기)

- 디컴파일 결과에서 FUN_xxxxxx, undefined4, PTR_FUN_xxx 같은 표기가 뭘 의미하는지
- xref(cross-reference) 개념 — "이 주소를 누가 참조하나" 추적하는 게 오늘 작업의 8할이었습니다
- C++ 바이너리 특유의 것들: vtable, RTTI, 네임 맹글링 (.?AVClassName@@ 같은 문자열 봤던 것들) — 이걸 알아야 클래스 구조를 추론할 수 있습니다

3. Windows 네트워킹 내부

- Winsock API: connect/send/recv vs WSAConnect/WSASend/WSARecv (오버랩드 I/O) 차이 — 오늘 브레이크포인트가 안 걸렸던 이유
- GetProcAddress 기반 동적 API 로딩이 왜 정적분석(임포트 테이블 검색)을 무력화시키는지
- netsh portproxy/IP Helper 서비스처럼 OS 레벨 리다이렉트가 어떻게 동작하는지

4. 네트워크 프로토콜 리버싱 방법론

- 패킷 캡처 방법론: 오늘처럼 가짜 서버를 만들어 실제 클라이언트와 주고받는 바이트를 관찰하는 게 정적분석보다 압도적으로 빠를 때가 많습니다
- 해시 함수 식별: 오늘 나온 Adler-32처럼, 디컴파일된 코드가 "이거 유명한 알고리즘이다"를 알아채는 감각 (상수 65521, NMAX 같은 게 힌트)
- 간단한 스트림 암호(XOR 롤링키) 패턴 인식

5. 라이브 디버깅

- WinDbg 계열 기본기: 브레이크포인트, 레지스터/스택 읽기, debugger_stack_trace류 개념
- 오늘처럼 정적분석이 막히면 라이브 디버깅으로 전환하는 판단력이 실전에서 제일 중요합니다
